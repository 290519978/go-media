# camera2 大屏交互说明

本文档用于统一 `dashboard/camera2` 的前端交互口径，避免大屏页面后续继续出现筛选、弹框和卡片行为不一致的问题。

## 1. 监控画面

- `MonitorGrid` 全屏必须继续运行在主应用树内，使用声明式覆盖层渲染，不再通过 `render(h())` 额外挂载独立 Vue 树。
- “区域 / 算法 / 设备状态”筛选保持 Ant Design 默认展开行为，不再手动维护显式 `open` 状态。
- 设备状态筛选固定为“在线 / 报警 / 全部”，由前端基于 `channels[].status` 和 `channels[].alarming_60s` 本地过滤，不再展示按设备逐条枚举的下拉项。
- 视频卡片右上角展示绑定算法，样式对齐 `camera` 大屏右上角算法胶囊，多算法按 `算法A / 算法B` 拼接，超长省略并保留完整 `title`。
- 当设备处于告警态时，右上角算法胶囊需要给告警角标让位，避免与红色告警图标重叠。
- 普通态筛选弹层挂在 `.app-container` 页面容器内，全屏态筛选弹层挂在 `.fullscreen-grid` 覆盖层容器内，避免 `screen-stage` 缩放层影响定位与遮罩交互。
- 全屏切换只改变展示容器，不重置筛选条件、搜索关键字、分页和宫格布局状态。
- `/dashboard` 入口默认使用应用内伪全屏覆盖层承载 `camera2`，不再额外提供前端内置的浏览器全屏开关；上传文件或其他系统交互不得导致退出当前大屏。

## 2. 卡片右下角操作

- `icon-setting` 只负责设备信息编辑，不再承载任务配置。
- 设备信息弹框按协议区分 RTSP、RTMP、GB28181：
  - 公共可编辑字段：设备名称、所属区域
  - RTSP：`origin_url`、`transport`
  - RTMP：`app`、`stream_id`、`publish_token`
  - GB28181：仅允许改公共字段，接入参数只读展示
- `icon-handler` 负责当前设备的“任务与算法快速编辑”，整体口径接近视频任务编辑，但只开放简化字段：
  - 任务名称
  - 录制策略：仅 `报警片段录制` / `不录制`
  - 算法绑定选择
  - 已选算法列表
- `icon-handler` 保留“新增算法”按钮，并接入真实业务链路：
  - 弹窗字段仅包含 `算法名称`、`提示词`
  - 新增算法固定按 `detect_mode=2`（仅大模型）创建
  - 算法编码由后端自动生成，页面不展示编码输入框
  - 创建成功后自动回填到当前父弹窗的已选算法列表，但仍需用户点击“保存设置”才会把绑定落库
- `icon-handler` 不直接编辑以下高级参数：
  - 抽帧模式、抽帧值
  - 报警录制前秒数、报警录制后秒数
  - 算法报警周期、报警等级
- 参数口径：
  - 已有任务里的当前设备：上述高级参数继续沿用原值
  - 当前设备新绑定的算法：报警周期、报警等级使用系统默认值
  - 当前设备未绑定任务时：创建仅包含当前设备的新任务，抽帧、录制前后秒、报警周期、报警等级均使用系统默认值
- 已选算法列表中的辅助说明统一展示为 `报警周期:X秒 | 报警等级:XXX`，不再显示旧的“检测频率 / 灵敏度”占位文案。
- `del-icon` 仅删除“当前设备上的算法绑定”，不删除算法定义本身。
- 新增算法弹窗内支持草稿测试：
  - 图片最多 5 张
  - 视频最多 1 个
  - 视频大小上限取 `config.toml -> Server.AI.AlgorithmTestVideoMaxBytes`
  - 草稿测试结果只在当前弹窗展示，不写入算法管理测试记录
  - 当前上传批次里的异常图片优先使用浏览器本地 `blob:` 预览地址显示并叠加画框，不依赖 `/api/v1/algorithms/test-media/*` 回显
  - 草稿测试图片画框与算法管理测试共用同一套 `normalized_boxes` 口径：`x/y` 为归一化中心点，`w/h` 为归一化宽高
  - 正常图片继续不展示图片卡片
  - 草稿测试图片分项命中可恢复失败时，会按 `config.toml -> Server.AI.AnalyzeImageFailureRetryCount` 统一补跑，`0` 表示关闭
- `camera2` 的“任务巡查”和右下角机器人输入框共用 `/api/v1/dashboard/camera2/patrol-jobs`
  - 巡查图片命中可恢复失败时，也会按 `config.toml -> Server.AI.AnalyzeImageFailureRetryCount` 统一补跑，避免两处入口行为不一致
- 上传按钮只允许本地选文件，不允许跳转到外部 `action` 地址；点击上传不得导致大屏退出或离开当前页面。

## 3. 保存后的运行时行为

- `icon-handler` 保存后需要自动处理当前设备的 AI 运行状态：
  - 当前设备已在运行：仅对当前设备执行 `stop + start`
  - 当前设备未运行：仅对当前设备执行 `start`
  - 同任务下其他设备不得被一起停掉或重启
- 若设备未绑定任务，则在大屏内创建一个仅包含当前设备的新任务，并在创建成功后立即启动。
- 大屏快速编辑只适合快速改绑算法和录制策略；抽帧、报警周期、报警等级、录制前后秒的精调仍在视频任务编辑页完成。

## 4. 报警处理详情

- AI 分析结果中的“分析结论”只允许展示 `llm_json.task_results[].reason` 解析结果。
- 优先按当前事件的 `algorithm_code -> task_results[].task_code` 精确匹配对应 `reason`。
- 若未匹配到当前算法，则回退到第一个 `alarm=1` 的 `reason`；仍没有时取第一个非空 `reason`。
- 若 `llm_json` 缺少可用 `reason`，界面统一显示“暂无分析结论”。
- 禁止把原始 `llm_json`、Markdown 代码块或序列化对象文本直接展示到“分析结论”区域。

## 5. `llm_json` 解析兼容

- 需要兼容标准 JSON。
- 需要兼容带有 ```json fenced block 的文本。
- 需要兼容被字符串再包一层的 JSON 文本。
- 解析失败时按“无结论”处理，不做原文透传。
