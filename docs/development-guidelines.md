# maas-box 开发规范

## 1. 适用范围

本规范适用于 `maas-box` 项目的后端、前端、AI 服务、配置文件和文档维护。

## 2. Windows 环境下的中文编码规范

项目当前主要在 Windows + Codex + IDEA 环境下协作，中文乱码是最高频问题之一。后续修改时必须遵守以下规则。

### 2.1 推荐方式

- 优先使用支持补丁式修改的方式改文件，避免整文件重写。
- 手工脚本写文件时，必须显式指定 `UTF-8` 编码。
- 修改完成后，立即复查：
  - 页面中文文案
  - Go/TS/Python 源码中的中文字符串
  - `toml/md` 文档中的中文内容

### 2.2 禁止方式

以下方式在 Windows 下容易把中文写坏，原则上禁止用于含中文文件：

- 未显式编码的 shell 重定向 `>`、`>>`
- 未显式编码的 `Set-Content`
- 未显式编码的 `Out-File`
- 在已乱码文件上继续追加局部修改

### 2.3 处理乱码的原则

- 一旦发现源码或文档已乱码，优先恢复到干净版本后重做修改。
- 不要在乱码文本上继续堆补丁，否则会显著拉长修复时间并提高引入语法错误的风险。

## 3. 中文注释规范

- 新增或修改复杂逻辑时，必须补充必要的中文注释。
- 重点场景包括：
  - 异步任务调度
  - 状态流转
  - 配置兜底
  - 录制/清理策略
  - AI/LLM 调用边界
  - 历史兼容代码清理点

### 3.1 注释要求

- 注释尽量简洁，说明“为什么”和“业务语义”。
- 不写无信息量注释，例如“给变量赋值”“调用函数”。
- 对外部依赖、时间窗口、并发限制、阈值策略等要写清楚用途。

## 4. 文档更新规范

代码变更不是交付完成，文档同步是默认要求。

### 4.1 必须更新文档的场景

- 新增或修改 API
- 新增或修改配置项
- 新增或修改部署方式
- 新增或修改任务流程
- 改变前端重要交互行为
- 改变 AI/LLM 提示词组装边界

### 4.2 文档放置建议

- 架构变化：更新 `docs/architecture.md`
- 部署与配置变化：更新 `docs/deployment.md`
- 阶段目标或范围变化：更新 `docs/development-plan.md`
- 专项规则：新增 `docs/*.md`

## 5. 新项目阶段的代码策略

当前 `maas-box` 处于新项目阶段，默认策略是不为旧方案保留额外兼容负担。

### 5.1 默认原则

- 优先删除无调用方的旧代码
- 优先合并重复实现
- 优先保留单一真实路径
- 不为了“可能以后会用到”保留未使用分支

### 5.2 适用示例

- 旧接口无调用方时，直接删除
- 旧提示词拼装逻辑无入口时，直接删除
- 旧任务链路被新实现替代后，直接清理

## 6. 交付前检查清单

每次改动完成后，至少确认以下项目：

- 中文未乱码
- 关键复杂逻辑已补中文注释
- 对应 `docs/` 已更新
- 无调用方旧代码已清理
- 已执行构建、测试或最小回归验证
