# maas-box 协作约定

本文件用于约束在 `maas-box` 仓库内进行代码修改、调试和文档更新时的默认行为。

## 1. 编码与中文文本

- 仓库内源码、配置、文档统一按 `UTF-8` 处理。
- 修改包含中文的文件时，优先使用不会破坏编码的编辑方式：
  - 首选：`apply_patch`
  - 次选：明确指定 `UTF-8` 编码的文件写入
- 在 Windows 环境中，禁止使用未显式声明编码的重定向或写文件方式处理含中文文件，例如：
  - `>`
  - `>>`
  - 默认编码的 `Set-Content`
  - 默认编码的 `Out-File`
- 如必须通过 shell 重写文件，必须显式使用 `UTF-8` 编码，并在修改后立即复查中文内容是否正常。
- 如发现文件已出现乱码，不要在乱码文件上持续局部修补；优先恢复到干净基线后重新应用改动。

## 2. 代码注释

- 新增或修改复杂逻辑时，需要补充简洁的中文注释，帮助后续排查和交接。
- 注释要求：
  - 解释“为什么这样做”或“这段逻辑的业务含义”
  - 不写无意义的逐行翻译式注释
  - 优先写在关键分支、状态机、异步任务、配置兜底和兼容性清理处

## 3. 文档同步

- 只要修改了以下内容之一，必须同步更新 `docs/` 下对应文档：
  - 接口
  - 配置项
  - 任务流程
  - 部署方式
  - 重要交互行为
- 如果没有现成文档承接，应新增文档，而不是只改代码不留说明。
- 功能完成后的最低要求：
  - 代码可运行
  - 测试或构建通过
  - 文档已更新

## 4. 新项目默认策略

- 当前项目属于新项目阶段，默认不以兼容旧实现为目标。
- 修改功能时，优先选择：
  - 删除无调用方的旧代码
  - 收敛重复逻辑
  - 统一单一实现路径
- 除非用户明确要求，否则不要为了“保守兼容”额外保留旧接口、旧分支、旧提示词拼装逻辑或旧任务链路。

## 5. 修改后的自检清单

完成修改后，至少检查以下事项：

- 中文内容是否正常，无乱码
- 是否补了必要的中文注释
- `docs/` 是否已同步
- 是否顺手删除了无用旧代码
- 是否执行了对应的构建、测试或最小回归验证
